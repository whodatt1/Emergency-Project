# EMERGENCY : 실시간 응급실 가용 병상 알림 서비스

## 프로젝트 개요
전국 응급실 및 입원실 가용 병상 정보를 2분 주기로 수집하고, 병원 정보 변동 시 **관심 병원을 등록한 유저에게 즉각적인 FCM 푸시 알림을 제공**하는 실시간 알림 서비스입니다. 

데이터 갱신과 알림 발송 과정에서 발생할 수 있는 병목 현상을 해소하고, 데이터가 유실되지 않도록 시스템 결합도를 낮추고 안정성을 확보하는 데 집중했습니다.

## 개발 환경
- **Backend:** Java 21, Spring Boot 3.x, MyBatis
- **Batch Processing:** Spring Batch, Spring Scheduler
- **Event Streaming:** Apache Kafka
- **Database:** MySQL
- **Push Notification:** Firebase Admin SDK (FCM)

---

## 시스템 파이프라인 흐름
데이터의 최신성 유지와 알림 발송의 안정성을 위해 아래와 같이 자동화된 흐름을 구축했습니다.

- **[Step 1] 데이터 수집:** Scheduler(2분 주기) → 공공데이터 API 호출 → DB 일괄 업데이트 (Bulk)
- **[Step 2] 변동 감지:** 업데이트 시각 변동 데이터 선별 → DB 트랜잭션 커밋 직후 `Spring Event` 발행
- **[Step 3] 비동기 분리:** 이벤트 핸들러 → `Kafka` 알림 Topic 전송 (시스템 간 물리적 격리)
- **[Step 4] 실시간 알림:** Kafka `Consumer` → 대상 유저 선별 → `FCM` 푸시 알림 전송

## 🚀 주요 성능 개선 및 트러블슈팅

### 1. Spring Batch 데이터 처리 성능 최적화
**(1) 적절한 PageSize 선택**
* **[문제]** API 호출 시 PageSize(100)가 너무 작아 불필요한 HTTP 네트워크 요청이 다수 발생.
* **[개선]** 한 번에 가져오는 데이터 단위를 전체 응답 건수에 맞춘 500건으로 늘려, 사이클 당 API 요청 횟수를 5회에서 1회로 축소.
* **[성과]** 네트워크 오버헤드가 감소하여 배치 수행 시간이 **8초 → 3초**로 유의미하게 단축됨.

**(2) ChunkSize 및 배치 반환 타입 수정**
* **[문제]** List 단위로 데이터를 통째로 처리하는 구조 탓에, 객체 변환 과정에서 불필요한 반복문과 메모리 오버헤드 발생.
* **[개선]** Chunk-oriented 아키텍처에 맞게 ChunkSize를 500으로 맞추고, Reader에서 Queue를 활용해 단일 객체 단위로 전달하도록 구조 개편.
* **[성과]** 데이터 평탄화 오버헤드가 제거되며, 전체 수행 시간이 **최대 2초**까지 추가 단축됨.

**(3) ModelMapper → MapStruct 전환**
* **[문제]** ModelMapper의 런타임 리플렉션 사용으로 인해 매핑 과정에서 성능 오버헤드 및 처리 시간 편차 발생.
* **[개선]** 컴파일 시점에 매핑 코드를 생성하여 단순 필드 대입만 수행하는 `MapStruct`로 라이브러리 교체.
* **[성과]** 들쑥날쑥하던 실행 시간이 **안정적인 2초대**로 유지되는 것을 확인.

### 2. Kafka 실시간 알림 시스템 도입 및 튜닝
**(1) DB와 Kafka 간 트랜잭션 정합성 분리**
* **[문제]** DB 업데이트와 Kafka 메시지 전송이 단일 트랜잭션에 묶여 있어, 알림 실패 시 DB까지 롤백되는 강결합 문제 존재.
* **[개선]** Spring Event(`AFTER_COMMIT`)를 활용하여 DB 업데이트 커밋이 완료된 이후 비동기로 Kafka 메시지를 전송하도록 분리.
* **[성과]** 알림 시스템 장애가 메인 비즈니스(DB 업데이트)에 영향을 주지 않는 **안전한 결합도 분리** 달성.

**(2) Consumer 실패 시 재시도 전략**
* **[문제]** Consumer 내부 예외 발생 시 안전장치가 없어, 일시적인 네트워크 오류에도 메시지가 영구 누락될 위험 존재.
* **[개선]** `DefaultErrorHandler`와 `FixedBackOff`를 적용하여 예외 발생 시 자동으로 최대 5회, 3초 간격으로 재시도하도록 설정.
* **[성과]** 일시적 오류로 인한 메시지 처리 **누락 방지 기반** 마련.

**(3) 기본 설정 튜닝으로 지연 시간 감소**
* **[문제]** Kafka Producer의 기본 설정 변경이 응답 지연 시간에 유의미한 영향이 있는지 확인 필요.
* **[개선]** 단일 브로커 환경에 맞춰 `linger.ms=0`, `acks=1`로 설정하고, 네트워크 대역폭 최적화를 위해 `snappy` 압축 옵션 적용.
* **[성과]** 평균 요청 지연 시간이 기존 300~600ms에서 **200~500ms 수준으로 약 20% 감소**.

### 3. 안정성 및 데이터 정합성 보장 (Outbox Pattern)
**(1) Outbox 패턴을 활용한 알림 메시지 유실 방지**
* **[문제]** Kafka 브로커 장애 시 증발해버린 메시지를 추적하거나 복구할 수 없음.
* **[개선]** 알림을 카프카로 바로 보내지 않고, 먼저 DB의 Outbox 테이블에 '발송 예정' 상태로 안전하게 저장해 둔 뒤 전송하도록 구조 변경.
* **[성과]** 알림 전송에 실패하더라도 DB에 실패 이력이 기록되어 언제든 재발송이 가능해져 **장애 상황에서도 대응 가능한 시스템** 구축.

**(2) 스케줄러와 낙관적 락(Optimistic Lock) 기반 실패 복구**
* **[문제]** 실패 건 재발송 스케줄러와 메인 배치가 동일 데이터를 동시에 수정할 경우 동시성 경합 및 정합성 문제 발생.
* **[개선]** `batch_id`를 활용한 낙관적 락을 도입하여, 스케줄러 선점 시 메인 배치가 이미 갱신한 데이터라면 재발송 업데이트가 무시되도록 처리.
* **[성과]** 유저에게 **중복된 알림이 발송되지 않도록 차단**.

**(3) UPSERT 적용으로 최신성 유지**
* **[문제]** 장애 복구 시, 동일 병원의 과거 실패 이력까지 한꺼번에 재발송되어 유저에게 잘못된 정보를 줄 위험 존재.
* **[개선]** Outbox 테이블에 Unique Index를 설정하고 `UPSERT` 쿼리를 적용하여, 동일 식별자는 항상 최신 정보로 덮어씌워지도록 설계.
* **[성과]** 실패 건이 쌓여도 재발송 시점에는 항상 **가장 최신 상태의 정보 1건만 발송**되도록 최적화.

### 4. 외부 의존성 격리 및 통합 테스트 환경 구축
**(1) 비동기 콜백(Outbox) 상태 동기화 검증**
* **[문제]** Kafka 등 외부 인프라가 띄워져 있어야만 비동기 처리 성공/실패 시 DB Outbox 상태가 제대로 업데이트되는지 테스트 가능.
* **[개선]** `@MockitoBean`으로 Kafka 요소를 격리하고 `CompletableFuture`를 조작하여 전송 성공/실패 상황을 인위적으로 연출하는 테스트 코드 작성.
* **[성과]** 외부 인프라 없이도 트랜잭션과 비동기 경계에서 발생하는 로직의 **신뢰성 확인 및 안정적인 테스트 환경** 구축.

**(2) Consumer 멱등성 및 대용량 청크 분할 발송 검증**
* **[문제]** 메시지 중복 수신 시의 알림 중복 발송 위험과, 대량의 FCM 토큰 조회 시 메모리 과부하를 막는 청크 처리의 정상 작동 여부 검증 필요.
* **[개선]** 가짜 Outbox 데이터를 세팅해 멱등성 방어(호출 0회)를 증명하고, Mocking을 통해 250개 토큰 발송 시 설정한 ChunkSize(100)에 맞춰 정확히 3번 분할 호출되는지 테스트.
* **[성과]** 중복 발송 장애와 대규모 알림 시 발생할 수 있는 메모리 문제를 사전 차단하여 **운영 환경 수준의 안정성 검증 완료**.

### 5. DB 쿼리 최적화
**(1) OFFSET 쿼리 제거 및 성능 개선**
* **[문제]** `OFFSET` 방식의 페이징은 데이터가 많아질수록 불필요한 스캔으로 인해 성능이 기하급수적으로 저하됨.
* **[개선]** 마지막 조회 식별자를 기준으로 삼는 **커서(Cursor) 기반 페이징** 방식으로 쿼리를 전면 개편.
* **[성과]** 대규모 데이터 환경에서도 풀 스캔을 방지하여 **일관되고 빠른 조회 속도를 보장**하는 기반 마련.

## 공통 리팩토링 및 코드 품질 개선
- **트랜잭션 최적화:** 단순 조회(SELECT) 계층의 불필요한 `@Transactional` 오버헤드 제거
- **예외 처리 규격화:** `GlobalExceptionHandler`를 도입하여 클라이언트 후처리에 용이한 공통 에러 응답 규격 제공
- **객체 지향 설계:** 생성자 주입 및 `final` 필드를 통한 싱글톤 빈 불변성 보장
- **타입 안전성 확보:** `.yml` 설정값을 `TypeSafeProperties` 객체로 관리하여 일관성 유지
- **DB Round-trip 최소화:** MyBatis `<foreach>`와 `IN` 절을 활용해 일괄 조회 방식으로 쿼리 호출 최소화
