# EMERGENCY : 실시간 응급실 가용 병상 알림 서비스

### 📖 프로젝트 개요
전국 응급실 및 입원실 가용 병상 정보를 2분 주기로 수집하고, 병상 수 변동 시 관심 병원을 등록한 유저에게 즉각적인 푸시 알림(FCM)을 제공하는 웹 서비스입니다. 
데이터 갱신과 알림 발송 과정에서 발생할 수 있는 병목 현상을 해소하고, 데이터가 유실되지 않도록 안정성을 확보하는 데 집중했습니다.

### 🛠 개발 환경
- **백엔드:** Java 21, Spring Boot 3.x, MyBatis
- **대용량 배치 처리:** Spring Batch, Spring Scheduler
- **비동기 메시지 처리:** Apache Kafka
- **데이터베이스:** MySQL
- **푸시 알림:** Firebase Admin SDK (FCM)

### 🔥 주요 성능 개선 및 트러블슈팅 요약
> 💡 상세한 기술적 고민과 아키텍처 설계 과정, 핵심 코드는 **[👉 상세 포트폴리오 문서 (Notion)](https://delicate-whip-9da.notion.site/Emergency-Project-32f3e1e2779f80dcae38e1b5f6f340b1?source=copy_link)**에서 확인하실 수 있습니다.

- **대규모 데이터 배치 처리 최적화 (수행 시간 8초 → 2초 단축)**
  - 한 번에 처리하는 데이터 단위를 최적화하고, 불필요한 반복문이 발생하던 구조를 개편하여 오버헤드를 축소했습니다.
- **시스템 결합도 분리 및 알림 지연 시간 20% 감소**
  - DB 업데이트와 알림 발송 트랜잭션을 분리하고, 비동기 메시지(Kafka) 설정을 튜닝하여 메인 비즈니스의 영향도를 없애고 처리 속도를 높였습니다.
- **장애 발생 시 알림 누락 방지 (Outbox 패턴)**
  - 서버 장애로 인한 메시지 증발을 막기 위해 DB에 임시 저장소를 두고, 유실 방지를 위한 재시도 스케줄러를 구현했습니다.
- **통합 테스트 환경 구축**
  - 실제 메시지 큐(Kafka) 환경 없이도 비동기 로직의 성공/실패 상황과 대용량 분할 발송의 안정성을 검증하는 테스트 코드를 작성했습니다.
- **쿼리 튜닝**
  - 데이터가 많아질수록 느려지는 페이징(OFFSET) 문제를 커서(Cursor) 기반으로 개편하여 안정적이고 빠른 조회 속도를 보장하도록 개선했습니다.

### 🔄 시스템 파이프라인 흐름
데이터의 최신성 유지와 알림 발송의 안정성을 위해 아래와 같이 자동화된 흐름을 구축했습니다.

1. **데이터 수집:** `Scheduler(2분 주기)` → 공공데이터 API 호출 → DB 일괄 업데이트
2. **변동 감지:** 병상 수 변동 데이터 선별 → DB 트랜잭션 커밋 직후 이벤트 발행
3. **비동기 분리:** 이벤트 핸들러 → 알림 대기열(Kafka) 전송 (시스템 간 물리적 격리)
4. **실시간 알림:** 알림 수신(Consumer) → 대상 유저 선별 → 푸시 알림(FCM) 전송
